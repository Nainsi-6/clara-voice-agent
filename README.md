# Clara Voice Agent Automation Pipeline

A deterministic pipeline that converts messy call transcripts into structured, versioned AI voice agent configurations. Built for reliability, traceability, and zero-cost operation.

---

## Quick Start

```bash
# 1. Clone the repository
git clone https://github.com/Nainsi-6/clara-voice-agent

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run the full pipeline for a single account
python quickstart.py

# 4. Or run batch processing across multiple accounts
python batch_processor.py ./sample_transcripts
```

All outputs are saved to `outputs/accounts/`.

---

## Overview

The Clara pipeline handles two types of sales/onboarding calls and converts them into versioned voice agent configurations:

| Call Type | Output | Purpose |
|-----------|--------|---------|
| **Demo Call** | `v1` | Initial exploratory agent configuration |
| **Onboarding Call** | `v2` | Confirmed, production-ready configuration |

`v2` is always generated by **patching v1** — never replacing it. This ensures full configuration history is preserved and every change is traceable.

If any field cannot be confidently extracted from a transcript, it is stored in `questions_or_unknowns` rather than guessed. This is a core design constraint.

## Pipeline Architecture

The pipeline is composed of four sequential stages:

```
Transcript Input (.txt)
        ↓
TranscriptExtractor
  └── Rule-based regex parsing
  └── Populates account memo fields
  └── Unmatched fields → questions_or_unknowns
        ↓
Account Memo (Structured JSON)
        ↓
Prompt Generator
  └── Converts memo into natural language system prompt
        ↓
Voice Agent Specification
  └── system_prompt
  └── routing_logic
  └── transfer_protocol
  └── fallback_protocol
        ↓
Versioned Output + Changelog
  └── outputs/accounts/{account_id}/v1/
  └── outputs/accounts/{account_id}/v2/
  └── changelog.json / changelog.txt
```

## Demo Call → v1 Flow

```
demo_transcript.txt
        ↓
extract_account_memo()         # Parse transcript, extract structured fields
        ↓
Account Memo v1                # Saved to v1/account_memo.json
        ↓
generate_agent_spec()          # Build full agent spec from memo
        ↓
Agent Spec v1                  # Saved to v1/agent_spec.json
        ↓
system_prompt.txt              # Saved to v1/system_prompt.txt
```

---

## Onboarding Call → v2 Flow

```
onboarding_transcript.txt
        ↓
extract_account_memo()         # Parse onboarding transcript
        ↓
Onboarding Delta               # Only changed/new fields extracted
        ↓
PatchApplier.apply_patch()     # Merge delta onto v1 memo
        ↓
Account Memo v2                # Saved to v2/account_memo.json
        ↓
generate_agent_spec()          # Regenerate agent spec from updated memo
        ↓
Agent Spec v2                  # Saved to v2/agent_spec.json
        ↓
Changelog Generated            # Saved to changelog.json + changelog.txt
```

## Output Artifacts

### `account_memo.json`

Structured JSON representation of all extracted business configuration for the account.

| Field | Description |
|-------|-------------|
| `company_name` | Client company name |
| `business_hours` | Operating hours and timezone |
| `services_supported` | List of supported service categories |
| `emergency_definition` | Criteria for emergency escalation |
| `routing_rules` | Call routing destinations and logic |
| `integration_constraints` | Known API or system limitations |
| `questions_or_unknowns` | Fields that could not be confidently extracted |

---

### `agent_spec.json`

Full voice agent configuration derived from the account memo. Includes:

- **System prompt** — agent persona, tone, and behavioral instructions
- **Routing logic** — intent-based call routing rules
- **Transfer protocol** — timeout values and escalation paths
- **Fallback protocol** — behavior when no routing rule matches

---

### `changelog.json` / `changelog.txt`

A structured record of every change applied when moving from v1 to v2.

Each changelog entry contains:

| Field | Description |
|-------|-------------|
| `field_name` | The configuration field that changed |
| `old_value` | Value from v1 |
| `new_value` | Updated value in v2 |
| `reason` | Why the change was made |
| `source` | Transcript that triggered the change |
| `timestamp` | ISO 8601 timestamp of when the change was recorded |

**Example changelog entry:**

```json
{
  "field_name": "call_transfer_timeout_seconds",
  "old_value": null,
  "new_value": 30,
  "reason": "Onboarding call clarifies transfer timeout requirement",
  "source": "onboarding_transcript",
  "timestamp": "2024-01-15T10:32:00Z"
}
```

## Extraction Strategy

All extraction is **pattern-based**. No LLM is used in the extraction layer — this ensures deterministic, reproducible results.

| Pattern Type | Regex / Pattern |
|---|---|
| Emergency — sprinkler | `sprinkler.*(leak\|burst)` |
| Emergency — fire | `fire.*alarm` |
| Emergency — pressure | `water.*pressure.*loss` |
| Emergency — system | `system.*offline` |
| Phone numbers | `\d{3}-\d{4}` |
| Business hours | `8 AM – 5 PM`, `Mon-Fri`, `Eastern Time` |

If no pattern matches, the field is added to `questions_or_unknowns` — it is **never guessed or inferred**.

---

## Design Principles

**No Hallucination**
Missing information is always recorded explicitly. The system never fills in gaps with assumptions or inferred values.

**Version Safety**
v1 outputs are never modified after creation. v2 is generated exclusively through patching, making the entire configuration history auditable.

**Deterministic Outputs**
Given the same transcript, the pipeline always produces the same structured result. There is no randomness or model variance in the extraction layer.

**Production Mindset**
The pipeline is built with idempotent execution (safe to re-run), structured logging at each stage, explicit error handling for missing fields, and a modular architecture where each component can be tested or replaced independently.
